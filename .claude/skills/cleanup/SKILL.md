# Cleanup — Find Leftover Dead Code

The user wants a sweep for things that became orphaned because of past changes: components nothing imports anymore, CSS classes the JSX no longer uses, redirects pointing at deleted pages, `TRIGGER_DISPLAY` entries no scenario uses, image files nothing references, dependencies nothing imports.

This is **not** a refactor pass and **not** the same as the built-in `simplify` skill (which reviews recently changed code for quality). This skill finds **dead code that survived changes** — the most common kind of cruft.

The user will say "cleanup", "find dead code", or "what's unused". They may also:
- Scope to a folder (`src/`, `static/`, `scenarios.ts`)
- Ask you to delete what you find
- Ask you to skip a specific check

---

## What you must do

### 1. Establish ground truth before deleting anything

Before reporting anything as "unused", read:
- `package.json` — to know the entry points (`docusaurus.config.ts`, `sidebars.ts`, scripts)
- `docusaurus.config.ts` — to see plugins, themes, redirects, navbar/footer items, customFields
- `sidebars.ts` — confirm the sidebar is autogenerated (so files in `docs/` are referenced by file system, not by sidebar config)
- `tsconfig.json` — to know which folders are excluded from compilation (`build`, `_context`, `.docusaurus`)

These tell you which files are "live entry points" — anything reachable from them is in use, and anything not reachable is a candidate.

### 2. Run the checks

#### A. Unused TypeScript/React exports in `src/`

Walk every `.ts` and `.tsx` file under `src/`. For each `export` (named, default, or re-export):
- Grep the rest of `src/` for an `import` that pulls that name
- Also check `docusaurus.config.ts`, `sidebars.ts`, `src/pages/` (Docusaurus discovers pages by file path), and `src/theme/` (Docusaurus theme overrides are discovered by name)
- If nothing imports it and it's not a Docusaurus auto-discovered file, flag it

**Special cases — do NOT flag:**
- Default exports of files under `src/pages/` — Docusaurus turns each into a route
- Default exports of files under `src/theme/` — Docusaurus uses theme swizzling by file name
- Anything imported via dynamic `import()` (grep for the filename in strings)
- Type-only exports used in `.d.ts` shims

#### B. Unused imports within files

Run `npm run typecheck` first. TypeScript with `noUnusedLocals` / `noUnusedParameters` will already catch most of these. If those flags are off in `tsconfig.json`, walk imports manually: for each `import { A, B, C }` or `import X`, grep for usage in the same file body (excluding the import line itself). Flag dead names.

#### C. Dead CSS classes

For each `*.module.css` under `src/`:
- Find the corresponding `.tsx`/`.ts` file (`styles.module.css` ↔ `index.tsx` in the same folder is the convention here)
- Parse out class names from the CSS (`.foo`, `.bar { ... }`)
- Grep the corresponding component for `styles.foo`, `styles['foo']`, or `clsx(styles.foo, ...)`
- Any CSS class not referenced is a candidate

For `src/css/custom.css` (global styles), the test is whether the class is referenced anywhere in `src/`, `docs/`, `blog/`, `static/`, or in any markdown that might use raw HTML. Be conservative — only flag classes that are clearly project-specific (e.g., `.scenarioCard__foo`), not Docusaurus theme overrides (`.navbar__brand`, `.theme-doc-sidebar-*`).

#### D. Stale entries in `src/data/scenarios.ts`

Read the file. Check:
- Every `TRIGGER_DISPLAY` key — if no scenario in `SCENARIOS` lists it in `triggerEvents`, flag the entry as orphan
- Every `ACTION_DISPLAY` key — if no scenario's `json.actions[*].name` (and any nested `params.actions[*].name`) uses it, flag
- Every `relatedScenarios` id — must exist as another scenario's `id`. If not, flag as broken reference (this is a real bug, not just dead code)
- Duplicate `id` values across scenarios → bug

#### E. Unreferenced redirects in `docusaurus.config.ts`

For each entry in the `@docusaurus/plugin-client-redirects` `redirects` array:
- The `to` path should resolve to a real doc — check that the corresponding file exists under `docs/`
- The `from` path is the historical URL; you can't verify it but you can flag if it's identical to `to` (no-op redirect)

#### F. Unreferenced static assets

For each file under `static/img/` (and any other `static/` subfolder):
- Grep `docs/`, `blog/`, `src/`, and `docusaurus.config.ts` for the filename (without the `static/` prefix, since Docusaurus serves `static/img/foo.png` as `/img/foo.png`)
- Also grep for `img/foo` (no extension) and the bare filename
- If nothing references it, flag

**Skip:**
- `favicon.svg`, `favicon.ico`, anything in `static/` referenced by `docusaurus.config.ts` `favicon` field
- Open Graph / social card images referenced by `themeConfig.image`
- Logo files referenced by `themeConfig.navbar.logo.src`

#### G. Unused npm dependencies

Two paths:
1. **If `depcheck` is installed**: run `npx depcheck` and merge results
2. **If not**: ask the user before installing. As a fallback, manually grep for each dependency name from `package.json` `dependencies` and `devDependencies` in `src/`, `docusaurus.config.ts`, `sidebars.ts`, `package.json` scripts, and `.github/workflows/`. Flag any name that has zero matches.

**Do not flag:**
- `@docusaurus/*` packages that are loaded by name string in plugins/themes/presets arrays in `docusaurus.config.ts` (e.g., `'classic'` resolves to `@docusaurus/preset-classic`)
- Type-only `@types/*` packages that pair with a runtime dependency
- TypeScript itself
- Anything referenced via npm scripts in `package.json`

#### H. Stale TODO / FIXME / XXX comments

Grep for `TODO`, `FIXME`, `XXX`, `HACK` across `src/`, `docs/`, `blog/`, and `_context/` is excluded. Report each with file path, line, and the comment text. These are not auto-deletable — they're surfaced for the user to triage.

#### I. Sidebar position conflicts

For every `_category_.json` under `docs/`, check that no two siblings in the same parent folder have the same `position` value. Conflicting positions cause unstable sidebar order. Report each pair.

For every `.md` file under `docs/`, check the same for `sidebar_position` in the frontmatter (siblings in the same folder).

#### J. Commented-out code blocks in `src/`

Look for blocks of 3+ consecutive `//` or `/* */` comment lines containing what looks like code (`import`, `function`, `return`, `const`, `<JSX>`, `}`). These are usually leftover from refactors. Report — do not auto-delete.

#### K. Empty or near-empty files

Files under `src/`, `docs/`, `blog/` that are 0 bytes or only frontmatter / only an `export {}` statement. These are usually leftover scaffolds.

Group by category, severity descending:

End with a one-line summary: `N candidates: X bugs, Y dead code, Z TODOs.`

### 4. Do not auto-delete unless asked

Default behavior: **report only**. Even if a file is clearly unused, do not delete it without confirmation.

If the user says "delete what's safe" or "clean it up":
- **Auto-deletable** (low risk):
  - Unused CSS classes (just remove the class block)
  - Unused imports (remove the import line, or the named symbol from a `{ ... }` import)
  - Unused entries in `TRIGGER_DISPLAY` / `ACTION_DISPLAY`
  - Unreferenced static assets (delete the file, but show paths first)
- **Always confirm before deleting** (higher risk):
  - Whole component files
  - npm dependencies (run `npm uninstall` only with explicit yes — it touches `package-lock.json`)
  - TODO comments (the user may want to keep them)
  - Anything that crosses a folder boundary

After any deletion, run `npm run build` and `npm run typecheck` to confirm nothing broke. If either fails, **revert the deletion** and report — better to leave the cruft than break the build.

### 5. False-positive escape hatches

If the user says "X is intentional, ignore it", remember it for this run. After the run, ask if they want it added to a permanent ignore list in this skill (e.g., a "do not flag" section in step 2).

## What NOT to do

- **Do not** delete files without explicit user confirmation. Even "clearly unused" files may be in-progress work the user hasn't wired up yet.
- **Do not** flag anything in `_context/` — that folder is a knowledge base, not source code, and is excluded from the build by `tsconfig.json`.
- **Do not** flag `Tutorial/` files as unused unless the user asks specifically — it's the default Docusaurus starter and the user may want to keep it as a reference. Mention it once, then leave it alone.
- **Do not** flag `node_modules`, `build/`, `.docusaurus/` — these are generated.
- **Do not** flag Hebrew or Arabic text as "unused strings" — they're production examples in scenarios.
- **Do not** uninstall a dependency without running `npm install` afterward to confirm the lockfile is consistent.
- **Do not** do anything destructive across multiple categories in one pass without showing the user the complete plan first.
- **Do not** treat TODO comments as "code to delete". They're surfaced for triage, not for automatic action.
- **Do not** "fix" sidebar position conflicts by renumbering — the user may prefer one ordering over the other. Report only.

## After the user confirms the cleanup is useful (or not)

Update this skill if:

- **A check produced a false positive the user dismissed** — narrow the rule, add to the "do not flag" list. Always include a one-line *why* so future-you doesn't re-add the false positive.
- **A check missed real cruft** — add the pattern. If you found dead code by hand that the skill should have caught, codify the heuristic.
- **A new "do not flag" exception** the user pointed out (a file Docusaurus discovers magically, a class used by a CMS-injected element, a dependency only used in a build hook) — add to the special-cases list in step 2.
- **A new category of cruft** the user wants scanned that isn't here — add a new check (3L, 3M, ...).
- **A category the user said is too noisy / never useful** — remove or convert to opt-in.
- **An auto-delete the user reverted** — move that category from "auto-deletable" to "always confirm" in step 4.
- **An auto-delete the user wished was applied without asking** — move it the other way (carefully).
- **A "remember for this run" exception** the user wants permanent — add to step 2's special cases or to "What NOT to do".
- **A scoping pattern the user used repeatedly** ("just src/", "just dependencies") — add a named mode and document the trigger phrase.
- **A new way the build broke after a cleanup** — add a guard to step 4 (e.g., "always check for `import()` dynamic imports before deleting an export").

Edit `.claude/skills/cleanup/SKILL.md` directly. Treat it as a living document. The most important thing this skill must learn over time is **what looks unused but isn't** — the special cases compound, and missing one corrupts the build.
