# Check Docs Consistency

The user wants every markdown file under `docs/` and `blog/` to follow the project's implicit conventions. This skill is a lint pass that finds drift before it ships.

The user will say "check the docs", "lint the markdown", or "are there any broken links in the docs". They may also:
- Scope to a folder or single file
- Ask you to fix what's safe to auto-fix
- Ask you to run only a specific check (links, frontmatter, etc.)

---

## What you must do

### 1. Establish the conventions before checking

Read these files first to ground the rules in what the project actually does, not what a generic linter expects:

- `docs/YAML/Overview.md` — top-level docs frontmatter
- `docs/YAML/Adapters/Plando/Plando.md` — gold-standard adapter doc, full of tables and tip callouts
- `docs/YAML/Types/Func/System/Request.md` — gold-standard function doc
- `blog/2026-04-02-plando.md` — gold-standard blog post
- `docusaurus.config.ts` — `onBrokenLinks`, `onBrokenMarkdownLinks` settings (currently `throw` and `warn`)
- `sidebars.ts` — confirms the sidebar is autogenerated (don't suggest manual edits)

### 2. Inventory the files

Glob `docs/**/*.md` and `blog/**/*.md`. **Note**: many files have spaces in their names (e.g., `Bot State Split.md`, `Check Working Hours.md`, `Send Mail.md`). Quote paths in shell commands. Do not "fix" the spaces — they are intentional and the URLs work.

Ignore:
- `_category_.json` files (not markdown)
- `node_modules/`
- `build/`
- `_context/` (not part of the site)
- `Tutorial/` (the unused Docusaurus starter)

### 3. Run the checks

#### A. Frontmatter

For every `docs/**/*.md` file:
- **Required**: `sidebar_position: <integer>` (sidebar is autogenerated; without this, ordering is alphabetical and usually wrong)
- **Allowed but not required**: `title: <string>`, `slug: </some/path>`
- **Forbidden**: anything else unless it's a known Docusaurus frontmatter key (`description`, `keywords`, `tags`, `image`, `pagination_prev`, `pagination_next`, `draft`, `unlisted`, `id`)

**Exceptions — do NOT flag missing `sidebar_position` on:**
- `docs/zMarkdown Examples.md` — markdown features cheat sheet. Uses the `z` filename prefix to force last-position alphabetical sort. No frontmatter by design. **Why:** user confirmed this is intentional and reflagging produces noise.
- `docs/API/API.md` — only file in its `docs/API/` folder, so per-file position is moot. **Why:** user confirmed this is intentional.

For every `blog/**/*.md` file:
- **Required**: `title`, `date`, `tags` (array of strings)
- The date in the frontmatter should match the date in the filename (`YYYY-MM-DD-*.md`)
- Tags should come from the established blog tag set (`docs`, `yaml`, `adapters`, `scenarios`, `automation`, `site`, `feedback`). Warn (don't error) on unknown tags.

For every blog post, check that `<!-- truncate -->` appears exactly once after the intro paragraph.

#### B. Internal links

Walk every `[text](url)` markdown link in every file. Classify:
- **Internal doc link**: starts with `/docs/`
- **Internal site link**: starts with `/scenarios`, `/changelog`, or `/`
- **Anchor**: starts with `#`
- **External**: starts with `http://` or `https://`
- **Mail/tel**: `mailto:`, `tel:`

For internal doc links:
- Resolve the path against `docs/` (replace `/docs/` prefix with `docs/`).
- The target file should exist as `<resolved>.md` OR a directory with an `index.md` / `<DirName>.md` inside it.
- URL-encoded spaces (`%20`) should resolve too — `docs/YAML/Types/Func/System/Send%20Mail` → `docs/YAML/Types/Func/System/Send Mail.md`.
- Anchor fragments (`#section-name`) are not validated against actual headings (too noisy) unless the user asks.

For internal site links (`/scenarios`, `/changelog`, etc.):
- Cross-check against the routes defined in `docusaurus.config.ts` `themeConfig.navbar.items` and the existence of files under `src/pages/`.

For external links: do **not** fetch them by default — too slow and flaky. The user can pass "check external links" to opt in.

Report every broken link with: file path, line number, link text, link href, and reason.

**Note**: `onBrokenLinks: 'throw'` in `docusaurus.config.ts` means the build already catches broken markdown links inside `.md` files. This skill exists to catch them **before** the build, and to also check things the build misses (e.g., links inside React components, links the build only warns about).

#### C. Code blocks

For every fenced code block (` ``` `):
- Allowed language tags: ` ```yaml `, ` ```json `, ` ```javascript `, ` ```bash `, ` ```typescript `, ` ```text `, ` ```diff `, ` ```md `, ` ```mermaid `.
- Unknown language tag → warn (the user may want to add it to the allowlist).

**Do NOT flag untagged blocks** (` ``` ` with no language). **Why:** A previous run of this skill produced 41 untagged-block warnings; the user dismissed all of them as non-issues. Most are deliberate syntax templates in `docs/YAML/Data Injection/{Overview,Providers,Transformers}.md` showing shapes like `%provider:path|transformer(args)%` or `|formatPhone(...)`. They aren't real YAML/JSON/text and tagging them as `text` would be cosmetic only. The rare "untagged YAML in a tip" case (e.g., the `attrFields:` snippet at the time in `Plando.md`) is rare enough to catch by eye when polishing — it's not what this lint pass is for.

For YAML blocks specifically (the most common in this project):
- Should be valid YAML if the user asks to deep-check. Default: skip the parse.
- Should use 2-space indentation, not tabs.

#### D. Heading structure

For every file:
- Exactly one `# H1` (the title). The build will warn if not.
- Headings should not skip levels (`##` → `####` is suspicious). Warn.
- The H1 should match the file's purpose — for an adapter doc, the H1 should be the CRM name. (Soft check — flag if the H1 is "Untitled" or empty.)

#### E. Admonition syntax

The project uses Docusaurus admonitions:
- `:::tip[Title]` ... `:::`
- `:::warning` ... `:::`
- `:::info` ... `:::`
- `:::note` ... `:::`

Check that every `:::` opener has a matching `:::` closer. Mismatched admonitions break MDX parsing.

#### F. Trailing whitespace, tabs, mixed line endings

- Warn on trailing whitespace at end of lines (auto-fixable) — **EXCEPT** lines ending in *exactly two* trailing spaces, which is the markdown hard-line-break syntax. Preserve those silently.
- Warn on tab characters in markdown body (auto-fixable to 2 spaces)
- Warn on mixed CRLF/LF line endings within a single file

**Why the 2-space exception:** lines like `Supported operations in code: ... \`closeTicket\`.  ` (Zoho.md:44) and the demo line in `docs/zMarkdown Examples.md:31` use the two-trailing-spaces hard break intentionally. Stripping them would silently change rendering. One trailing space, three+ trailing spaces, and tabs are still flagged.

These are cosmetic but easy to fix and they show up in diffs.

### 4. Optional: defer to `markdownlint-cli2`

If `markdownlint-cli2` is installed, you can also run it for the structural checks (`MD001` through `MD059`). It catches things this skill doesn't (e.g., line length, list indent consistency, ordered-list numbering).

Check if it's installed:
```bash
npx --no-install markdownlint-cli2 --version 2>/dev/null || echo "missing"
```

If missing, **ask the user** before installing — it's a dev dependency. If they say yes:
```bash
npm install --save-dev markdownlint-cli2
```

If installed, run it scoped to `docs/**/*.md` and `blog/**/*.md` and merge results into the report. Suggested config disables a few rules that conflict with this project:
- `MD013` (line length) — off, the project uses long lines in tables
- `MD033` (no inline HTML) — off, the project uses `<br/>` and `&lt;`
- `MD041` (first line h1) — off, frontmatter comes first

If a `.markdownlint-cli2.jsonc` file exists at the repo root, use it as-is and don't override.

### 5. Auto-fix mode

By default, **report only**.

If the user says "fix" or "auto-fix", apply only the safe fixes:
- Strip trailing whitespace **except** lines ending in exactly two trailing spaces (markdown hard line break — preserve)
- Convert tabs in markdown body to 2 spaces (skip code blocks)
- Normalize line endings to LF

**Do NOT auto-add language tags to untagged code blocks** — that check has been removed entirely (see 3C). The user dismissed all 41 untagged-block findings as non-issues.

Do **not** auto-fix:
- Missing frontmatter (you don't know the right `sidebar_position`)
- Broken links (you don't know the intended target)
- Heading structure (could break the page)
- Anything in a code block (the user may have intentional formatting)

After auto-fixing, run `npm run build` to confirm nothing broke, then report what was changed.

### 7. Scoping

- **"check docs/YAML/Adapters"** → Glob only that subtree
- **"check blog"** → Glob only `blog/`
- **"check the new Plando file"** → Glob the single file
- **"frontmatter only"** → run only check 3A
- **"links only"** → run only check 3B

## What NOT to do

- **Do not** rename files to remove spaces. The space-in-filename convention is intentional and the URLs work.
- **Do not** edit `sidebars.ts`. The sidebar is autogenerated.
- **Do not** rewrite `description`, `title`, or body content for "clarity" — that's content, not lint.
- **Do not** flag Hebrew or Arabic text as "non-English". It's intentional production-text examples.
- **Do not** treat warnings as failures. The user may consciously be deferring something.
- **Do not** validate external link targets by default. Only fetch them if the user explicitly opts in.
- **Do not** install `markdownlint-cli2` without asking.

## After the user confirms the lint is useful (or not)

Once the user has triaged your findings, update this skill if:

- **A check produced false positives** the user doesn't want — relax the rule, narrow the pattern, or convert error → warning. Add a one-line "why" so future-you doesn't re-tighten it.
- **A check missed real breakage** — add the pattern to step 3.
- **A new known frontmatter key** the user uses — add it to step 3A's allowlist.
- **A new blog tag** that should be considered established — add it to step 3A.
- **A code-block language tag** the user uses regularly that you flagged — add it to step 3C's allowlist.
- **An admonition variant** the user uses (e.g., `:::caution`) — add it to step 3E.
- **A file or folder pattern to ignore** that's not currently ignored — add to step 2.
- **An auto-fix the user said was unsafe** — remove it from step 5.
- **An auto-fix the user wished you applied** — add it to step 5.
- **A reporting format detail** the user prefers — update step 6.
- **A new tool worth integrating** (e.g., `mdformat`, `vale`) — add a section like step 4 with install gating.

Edit `.claude/skills/check-docs/SKILL.md` directly. Treat it as a living document. The goal is the noise floor drops every run.
